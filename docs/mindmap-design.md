# Mindmap as the Agent's Memory System - Design Doc

This doc provides structural information of the `mindmap`, an agent memory system.

## What is a Mindmap

The mindmap is a tree structure of knowledge. The simplest form is a single `node` as the root level,
and more nodes can "branch out" from the root as the first level, then the second level, third level...

```
        root
      /   |   \
  topic1 topic2 topic3 -- level 1
        / |          
    topic 2.1 topic 2.2 -- level 2
```

To query the knowledge, one should start from the root and go down the tree.

Nodes are connected in two ways:
1. through parent-child relationship
2. via links

A link is a directional arrow from a node (source) to another node or `resource` (target). 
The target node does not need to be at the same level as the source node.
The resources are usually files and can only be the target, not source.

Notice this structure has an intuitive mapping with the markdown file, so we
intend to `compile` a markdown file into a mindmap.

**Furthermore, a mindmap can serve as a persistent memory for the LLM agent**.

## Type Declarations

> Note that the below type declarations are not exhaustive; more details to be added.

```ts
interface Node {
    /** the slash separated "logic path" to a node, starting from root ('/')
     * e.g. "/skill/example/anti-pattern"
     */
    id: string
    /** the original text of the markdown section */
    text: string
    /** the node's title */
    title: string
    /** the node's summary, generated by LLM using a well-defined process */
    summary: string
    /** what level this node is at */
    level: number
    /** direct child nodes */
    children: Node[]
    /** outward links to other resources */
    links: Link[]
}

interface Mindmap {
    /** the backed resource folder */
    dir: string
    /** hash of the original markdown */
    hash: string
    /** time of compilation */
    compiled_at: Date
    /** time of modification */
    updated_at: Date
    /** root node */
    root: Node
}

interface Link {
    // NOTE: consider using union types
    target_type: 'node' | 'file' | 'url'
    node_id?: string
    file_path?: string
    url?: string
    /** the comment of this relationship */
    comment: string
}
```

## Utility Functions

Several util functions help to interact with the markdown or the mindmap.

### 1. compile_mindmap

Compile a markdown file into JSON with a backed folder as resource container.
Returns the compiled JSON which can be loaded as a mindmap.

The compilation process is detailed in the "Mindmap building algorithm" section.

The signature of `compile_mindmap` is:
```
compile_mindmap(md: filePath, cwd?: dirPath) -> JSON
```

### 2. patch_mindmap

Given a mindmap, update a node's text with optional feedback and trigger cascading updates
onwards/downwards.
Returns the information of the targeting node.

The signature of `patch_mindmap` is:
```
patch_mindmap(mindmap: Mindmap, id: string, text: string, feedback?: string) -> Node
```

### 3. summarize_node

Given a mindmap, generate the summary of a specific node without affecting other nodes.
Returns the summary of the targeting node.

The signature of `summarize_node` is:
```
summarize_node(mindmap: Mindmap, id: string) -> string
```

### 4. get_node

Given a mindmap, get a node's information by its path(id).
Returns the node's information.

The signature of `get_node` is:
```
get_node(mindmap: Mindmap, id: string) -> Node
```

### 5. validate_mindmap

Check if a JSON-as-mindmap is compatible with a markdown file.
Returns true/false.

Note that the "backed resource folder" should be recorded inside the json, 
so no need to pass in as an argument.

The signature of `validate_mindmap` is:
```
validate_mindmap(json: JSON, md: filePath) -> boolean
```

### 6. load_mindmap

Load a JSON and get a mindmap instance. The JSON should be first validated then loaded.

The signature of `load_mindmap` is:
```
load_mindmap(json: JSON) -> Mindmap
```

## Mindmap Building Algorithm

**To build a mindmap**, you take a markdown file and a (optional) folder as input.

First, parse the markdown file into a tree structure according to the markdown headings.
H1 headers correspond to level 1 nodes, H2 headers correspond to level 2 nodes, and so on.
Root node is generated from the markdown file, not corresponding to any part of markdown.

Second, summarize each node in a bottom-up order. Each node should use `A-N-C-E`(detailed below)
as the context, and the summarizing is done recursively.

Lastly, fix the metadata recorded in the Mindmap properties, and return the JSON.

**When patching a node**, first update the node's text inside the mindmap,
then update all the summaries of its descendants (direct or indirect) from the bottom up,
then update the summary of itself and all its ancestors up to the root node.
Finally, fix the metadata in the Mindmap, then return the summary of the targeting node.

### A-N-C-E Context Algorithm

To summarize a node, you must prepare the context. The context consists of four parts:

| Component | Description | Purpose |
|-----------|-------------|---------|
| **A** (Ancestors) | Combined text of all ancestor nodes | Provides upward context - where this node fits in the hierarchy |
| **N** (Node) | The text of the node itself | The actual content to summarize |
| **C** (Children) | Combined summaries of all descendant nodes | Provides downward context - what details exist below |
| **E** (Environment) | Agent behavioral guidance | Instructions for how to summarize (similar to PLAN mode) |

#### A - Ancestor Texts

The ancestor texts provide the "path context" - understanding where this node sits in the overall structure.

Example: For node `/skill/example/anti-pattern`, the ancestor texts would be:
- Root node text (project overview)
- `/skill` node text (skills section)
- `/skill/example` node text (examples section)

This gives the summarizer context about the hierarchical scope.

#### N - Node Text

The actual markdown content of the section being summarized. This is the primary input.

#### C - Descendant Summaries

All summaries of direct children combined. Since summarization is bottom-up, children are summarized before parents.

This provides "what's inside" context - the summarizer knows what details exist without needing to include them all.

#### E - Environment (Agent Behavior)

The behavioral guidance for the summarization task, similar to PLAN mode instructions. This includes:
- **Purpose**: Why we're summarizing (e.g., "for LLM context retrieval")
- **Constraints**: Length limits, style requirements
- **Focus areas**: What aspects to emphasize or de-emphasize
- **Output format**: How the summary should be structured

Example E instruction:
```
Summarize this node for an LLM agent's context retrieval.
- Keep summaries concise (50-100 words for leaf nodes, 20-50 words for branch nodes)
- Preserve key technical terms and identifiers
- Include important relationships and dependencies
- Omit boilerplate and redundant explanations
```

## Link Stub Design

Links are designed as **stubs for extensibility** - the basic structure is defined but full resolution is not yet implemented.

### Current State

Links are stored in the `Link` interface with:
- `target_type`: 'node' | 'file' | 'url'
- Target reference fields (`node_id`, `file_path`, `url`)
- `comment`: Description of the relationship

### Intended Behavior

Links allow nodes to reference external resources or other nodes:

```
[Node A] --link--> [Node B]     (node-to-node cross-reference)
[Node A] --link--> [file.rs]    (node-to-file reference)
[Node A] --link--> [URL]        (node-to-external reference)
```

### Future Extensions

The stub design allows for:
1. **Link resolution**: Following links to get referenced content
2. **Link validation**: Checking if targets exist
3. **Link traversal**: Walking the link graph
4. **Bidirectional links**: Tracking which nodes link to a given node

### Note on Markdown Parsing

Links in markdown (both inline and reference-style) would be parsed during compilation:
```markdown
See [related concept](./other.md) for more details.
```
Would create a Link with `target_type: 'file'` and `file_path: './other.md'`.

## Process Isolation

Each agent process has its own independent mindmap instance. This is critical for the multi-agent architecture.

### Isolation Boundaries

| Process | Mindmap Instance | Storage Location |
|---------|------------------|------------------|
| Lead (main) | Independent | `.mycc/mindmap.json` |
| Teammate A | Independent | `.mycc/mindmap-{name}.json` |
| Teammate B | Independent | `.mycc/mindmap-{name}.json` |

### Why Isolation Matters

1. **Different Perspectives**: Each agent may need different knowledge summaries
2. **No Race Conditions**: Updates don't conflict between processes
3. **Clean Shutdown**: Agent failures don't corrupt other agents' state
4. **Worktree Independence**: Teammates working in different worktrees have separate contexts

### Cross-Process Communication

If knowledge sharing is needed between agents:
- Use the **Wiki** (RAG) for shared persistent knowledge
- Use **mail_to** for point-to-point communication
- Mindmap remains process-local

### Wiki vs Mindmap

| Aspect | Wiki | Mindmap |
|--------|------|---------|
| Scope | Project-level, shared | Process-level, isolated |
| Storage | Vector database | JSON file |
| Query | Semantic search | Path traversal (get_node) |
| Use case | General knowledge retrieval | CLAUDE.md navigation |
| Updates | Manual (wiki_put) | Compilation from markdown |

## Compilation Flow

The mindmap compilation is **uni-directional**:

```
CLAUDE.md --> [compile_mindmap] --> mindmap.json
```

### Key Points

1. **Source of Truth**: `CLAUDE.md` is the authoritative source
2. **Derived Artifact**: `mindmap.json` is generated, not manually edited
3. **Re-compilation**: When CLAUDE.md changes, re-run `/mindmap compile`

### Startup Behavior

```
┌─────────────────────────────────────────────────────────────┐
│                      mycc start                              │
├─────────────────────────────────────────────────────────────┤
│  1. Check for .mycc/mindmap.json                            │
│     ├─ Exists? ──> Validate hash against CLAUDE.md           │
│     │              ├─ Valid? ──> Load mindmap               │
│     │              └─ Invalid? ──> Warn user, use fallback │
│     └─ Missing? ──> Warn user to run /mindmap compile       │
│                                                              │
│  2. Prepare agent context                                   │
│     ├─ Load root node summary                               │
│     └─ Add "use get_node tool" instruction                 │
│                                                              │
│  3. Agent uses get_node to navigate knowledge on-demand     │
└─────────────────────────────────────────────────────────────┘
```

### Fallback Behavior

If mindmap validation fails at startup:
- Agent receives minimal context (just root node if available)
- Warning message suggests running `/mindmap compile`
- Agent continues but may lack full project knowledge

## Workflows

The mindmap is originally designed for better understanding of `CLAUDE.md` but the process
has been generalized since then. For `mycc`, the mindmap is used to replace the initial
context population of `CLAUDE.md` -- previously the file content, now only the root node
and a piece of notice telling LLM to `get_node` on its own if needed.

The expected workflow of user/mycc interaction is:
- During mycc's start, the mindmap is loaded by reading `./.mycc/mindmap.json` and load the json.
- If in the previous step the validation failed, a warning is shown to advise the user to 
use `/mindmap compile` to re-compile. The re-compilation is not done automatically by mycc.
- The user sends a query to mycc
- Mycc finds itself lacking of knowledge, and initiates the knowledge searching using tools, including `get_node`
- After getting enough background knowledge, mycc is good to work.
- During the work if there is misunderstanding and the user has pointed it out, patch the mindmap
without touching the CLAUDE.md; only when the user asks mycc to write to CLAUDE.md should
it be updated (and re-compiled).
