# 漫游指南：计划单人模式

> *一篇 mycc 代理循环的漫游指南——在计划模式下，代理只规划，不碰代码。*

---

## 序幕：进入计划模式

用户想在动手之前先探索。他们输入：

```
agent >> /mode plan
```

**SLASH** 状态拦截了这个命令——LLM 看不到它。`/mode` 命令被路由到处理器，它：

1. 将模式设置为 `'plan'`
2. 将提示符从 `agent >>`（黄底黑字）改为 `plan >>`（蓝底白字）
3. 将系统提示词切换为计划模式版本

用户也可以通过以下方式进入计划模式：
- `/plan on`——专用的斜杠命令
- `plan_on` 工具（由 LLM 自己调用，带有可选的 `allowed_file` 参数）

现在提示符显示：

```
plan >> 我需要先了解项目结构，然后再做修改
```

---

## 第一幕：计划模式的系统提示词

当 LLM 在计划模式下被调用时，它收到一份不同的系统提示词。

### 有什么不同

| 方面 | 普通模式 | 计划模式 |
|--------|----------|-----------|
| **提示前缀** | `agent >>`（黄底黑字） | `plan >>`（蓝底白字） |
| **使命** | "使用工具完成任务" | "产出一个清晰、可执行的计划" |
| **代码修改** | 允许 | **禁止** |
| **工具访问** | 全部 30+ 工具 | 只读子集 |
| **Bash 命令** | 完整执行 | 仅 READ/TEST 动词 |
| **困惑评分** | 标准 | 每轮助手回合 +1 |
| **LLM 思考** | `think: false` | `think: true` |

### 计划模式指令

系统提示词告诉 LLM：

> *你处于**计划模式**。你的目标不是实现功能，而是：*
> 1. *通过探索代码库彻底理解问题*
> 2. *与用户澄清假设和歧义*
> 3. *产出一个**单一、清晰、可执行**的计划，包含具体的实现步骤*

它还包含一个特定的退出流程：

> *1. **先展示你的计划**——不使用任何工具结束当前回合*
> *2. **然后使用 plan_off**——在用户确认你的计划后*

### `allowed_file` 特性

`plan_on` 工具支持一个 `allowed_file` 参数。如果指定了，系统会询问用户：

```
是否允许在计划模式下编辑此文件？
  - 按 Enter 或输入 'y'/'yes' 允许此文件
  - 输入 'n'/'no' 进入严格计划模式（不允许任何文件）
  - 或输入其他文件路径来允许编辑那个文件
```

这在需要更新规划文档（如 `docs/plan.md`）时很有用。

## 第二幕：COLLECT——相同管道，更快困惑

COLLECT 状态运行与普通模式相同的管道：
- 邮件收集（单人模式下通常为空）
- 提示轮次检查（困惑值 ≥ 10）
- Todo 提醒（每 3 轮）
- Brief 提醒（每 5 轮）
- 主动技能发现

但**困惑指数上升得更快**。在 HOOK 状态中，计划模式下每轮助手回合都会增加 **+1** 困惑值：

```typescript
// 来自 hook.ts —— 困惑评分
if (ctx.core.getMode() === 'plan') {
  ctx.core.increaseConfusionIndex(1);
}
```

在计划模式下，代理无法采取生产性行动（写文件、运行构建）。它只能阅读和探索。系统加速了困惑时钟，使得**提示轮次触发得更早**——代理能更频繁地获得指导。

---

## 第三幕：LLM——探索模式

LLM 收到计划模式的系统提示词后，调整了自己的行为。

### LLM 能做什么

LLM 不写代码，而是专注于探索：

1. **读取文件** 使用 `read_file` 理解现有代码
2. **搜索** 使用 `grep` 查找模式和引用
3. **探索** mindmap 使用 `recall` 获取项目知识
4. **搜索网络** 使用 `web_search` 查找文档
5. **创建 todo** 使用 `todo_create` 跟踪规划项
6. **使用 `plan_on`** 进入带有允许文件的计划模式

### LLM 不能做什么

以下操作在计划模式下**被阻止**：

| 操作 | 工具 | 结果 |
|-----------|------|-------------|
| 写文件 | `write_file` | 授权系统拒绝："计划模式已激活" |
| 编辑文件 | `edit_file` | 授权系统拒绝："计划模式已激活" |
| 破坏性 bash | 带有 WRITE/EDIT/DELETE/BUILD/INSTALL 的 `bash` | Bash 裁判拒绝："不能在计划模式下 WRITE" |
| 交互终端 | `hand_over` | 被阻止 |

### 十字路口仍然工作

**十字路口**系统在计划模式下仍然运作。如果 LLM 的回复包含转折词如"然而"或"但是"，系统会生成替代续写。这在计划模式下很有用，因为代理正在探索多种方法——十字路口帮助它决定进一步调查哪条路径。

### LLM 思考模式

在计划模式下，LLM 调用使用 `think: true`，这启用了模型的内部推理（思维链），然后再产生回复。

## 第四幕：HOOK——计划模式执行者

HOOK 状态是计划模式限制得到强制执行的地方。

### 困惑评分

HOOK 状态在计划模式下每轮助手回合增加 **+1 困惑值**。这是确保在长时间探索过程中提示轮次能触发的核心机制。

### Hook 条件

通常会在代码变更时触发的 hook（如 `lint-after-edit`、`test-after-edit`）在计划模式下是**非活跃的**——没有东西可以提交。需要检查计划模式的 hook 可以在条件中使用 `seq.isPlanMode()`。

### 授权系统的 5 步评判

**5 步 bash 评判过程**是核心执行机制。在计划模式下，第 4 步变成了一个硬性关卡：

1. **危险模式**：相同检查（rm -rf、sudo 等）——不变
2. **意图语法**：相同验证——不变
3. **缺失意图**：相同检查——不变
4. **模式 + 动词检查**：如果模式是 `plan` 且动词是 `WRITE`、`EDIT`、`DELETE`、`BUILD` 或 `INSTALL`，命令被**立即拒绝**，提示：
   > `不能在计划模式下 WRITE。动词 "WRITE" 会修改状态。请先切换到普通模式，或使用只读动词如 [READ] 或 [TEST]。`
5. **LLM 分析**：仅针对计划模式下的 `RUN` 动词——LLM 判断命令是只读的还是修改状态的
6. **用户提示**：仅针对不确定的 `RUN` 命令

对于文件操作（`write_file`、`edit_file`），**授权评估器**会单独检查计划模式：

```typescript
if (mode === 'plan') {
  const allowedFile = core.getAllowedFile();
  if (request.path && allowedFile && resolvedRequested === resolvedAllowed) {
    return { approved: true }; // 允许文件例外
  }
  return { approved: false, reason: '计划模式已激活——代码修改暂时受限。' };
}
```

## 第五幕：TOOL——只读执行

计划模式下的工具执行很直接——只有只读工具能成功。

### 什么能工作

```typescript
// 读取文件——工作
read_file({ path: "src/math.ts" })

// 搜索——工作
grep({ pattern: "fibonacci" })

// 知识——工作
recall({ path: "/" })
wiki_get({ query: "project structure", domain: "project" })

// 规划——工作
todo_create({ name: "规划斐波那契实现" })
issue_create({ title: "添加斐波那契", content: "..." })

// 只读 bash——工作（使用 READ 动词）
bash({ command: "ls src/", intent: "READ SOURCE TO explore structure" })
```

### 什么会失败

```typescript
// 写入——被阻止
write_file({ path: "src/math.ts", content: "..." })
// 错误：计划模式：禁止修改代码

// 编辑——被阻止
edit_file({ path: "src/math.ts", old_text: "...", new_text: "..." })
// 错误：计划模式：禁止修改代码

// 破坏性 bash——被阻止
bash({ command: "npm install lodash", intent: "INSTALL DEPENDENCY TO add lodash" })
// 错误：不能在计划模式下 INSTALL。动词 "INSTALL" 会修改状态。
```

### TOOL 状态中的困惑评分

TOOL 状态也会贡献困惑评分：

- **探索工具**（`read_file`、`grep`、`web_search` 等）：困惑值不变
- **行动工具**（`todo_create`、`issue_create` 等）：减少困惑（有进展）
- **重复操作**：反复使用同一工具 → +1 困惑
- **错误**：工具返回错误 → +2 困惑

在计划模式下，由于代理主要使用探索工具，困惑值不像普通模式那样容易下降。再加上 HOOK 状态每轮 +1 的加成，困惑指数稳步上升，确保提示轮次能定期触发。

## 第六幕：STOP——规划完成

STOP 状态正常运行：
- **信函框**以绿色边框显示 LLM 的最终分析
- **团队等待**在单人模式下无操作
- 机器返回 PROMPT

但提示符仍然是 `plan >>`——代理保持在计划模式，直到明确退出。

---

## 退出计划模式

用户可以通过以下方式退出计划模式：
- `/mode normal`——斜杠命令（立即生效，无需确认）
- `/plan off`——上述命令的别名
- `plan_off` 工具（由 LLM 调用）

### 确认守卫

当 LLM 使用 `plan_off` 时，需要**明确的用户确认** `[y/N]`：

```
退出计划模式并允许代码修改？[y/N]
```

只有 `y` 或 `yes` 授予权限。任何其他回复（包括按 Enter）都会让代理保持在计划模式。这防止代理意外切换模式并在用户不知情的情况下做出修改。

`plan_off` 工具是**幂等的**——在普通模式下调用它只会返回成功，不会弹出提示。

### 两步退出协议

计划模式的系统提示词教会 LLM 一个特定的退出流程：

1. **先展示你的计划**——不使用任何工具结束当前回合，呈现完整计划
2. **然后使用 plan_off**——在用户确认计划后，请求退出

这确保用户在代理开始实现之前，总是能看到并批准计划。

## 对话记录：一个真实的计划模式会话

以下是一个典型的计划模式对话：

```
agent >> /mode plan
[提示符变为蓝色：plan >>]

plan >> 我需要添加一个斐波那契函数。让我先探索一下。

plan >> read_file({ path: "src/math.ts" })
[系统读取文件]

plan >> grep({ pattern: "fibonacci" })
[系统搜索——没有结果]

plan >> 让我看看项目结构。

plan >> bash({ command: "ls src/", intent: "READ SOURCE TO explore structure" })
[系统列出文件]

plan >> 我看到了项目结构。这是我的计划：

我将在 src/math.ts 中添加一个斐波那契函数。实现步骤：
1. 添加 `fibonacci(n: number): number` 函数
2. 处理边界情况（n <= 0, n === 1）
3. 从模块中导出该函数

我可以按这个计划执行吗？

plan >> plan_off()
[系统询问：] 退出计划模式并允许代码修改？[y/N]
[用户输入：] y

[提示符变为黄色：agent >>]
agent >> 好的，让我来实现斐波那契函数。
```

## 尾声：何时使用计划模式

计划模式适用于：

- **代码库探索**：在做出修改前理解不熟悉的代码
- **架构规划**：在实现前设计解决方案
- **调试调查**：读取日志、追踪代码路径
- **文档审查**：阅读现有文档而不编辑
- **教学场景**：当用户想学习而不让代理做出修改时

通过移除写代码的能力，代理专注于理解、分析和规划。更快的困惑时钟确保代理不会陷入无休止的探索循环。

## 状态机图

计划模式下运行相同的 6 状态机，每个状态有特定于计划模式的行为：

```
        ┌────────────────────────────────────────────┐
        │                                            │
   ┌─── PROMPT ◄────────────────────┐               │
   │    │   ▲                       │               │
   │    ▼   │                       │               │
   │  SLASH─┘                       │               │
   │                                │               │
   │    ▼                           │               │
   │  COLLECT ◄─────── TOOL ─────┐ │               │
   │    │              ▲         │ │               │
   │    ▼              │         │ │               │
   │  LLM ────► HOOK ──┘       STOP ──────────────┘
   │                │              │           │
   │          has calls        no calls    has mail
   │                                        or question
   └── (pendingSlashQuery set by SLASH)
```

计划模式下的差异：
- **PROMPT**：显示 `plan >>`（蓝底白字）而非 `agent >>`
- **SLASH**：`/mode plan` 和 `/plan on` 进入计划模式；`/mode normal` 和 `/plan off` 退出
- **COLLECT**：相同管道，但困惑上升更快
- **LLM**：使用计划模式系统提示词，`think: true`
- **HOOK**：授权系统拒绝 WRITE/EDIT/DELETE/BUILD/INSTALL 动词
- **TOOL**：只有只读工具成功
- **STOP**：正常，但提示符保持 `plan >>`
